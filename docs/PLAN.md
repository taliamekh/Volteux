## Problem Statement

Beginners who buy an Arduino starter kit fall off a cliff. They don't know what each component is, they don't know which pin to wire, they don't know how to write the C++, and the first compile error sends them straight to a YouTube tutorial that's two boards out of date. The existing in-browser tools (Wokwi, Tinkercad Circuits, Arduino Cloud IDE) all assume you already have the hardware AND already know what each piece does. They're simulators for people who already crossed the gap.

Volteux closes the gap. Beginner types a project description in plain English. AI picks the right archetype, writes a sketch, lays out a breadboard, and pre-fills an Adafruit cart you click to buy. While the parts ship, you click around an interactive 3D view that labels each component and explains what it does. When the kit arrives, you wire it up against the breadboard view and click "flash" — Chrome pushes the firmware to your board over USB.

The wedge is the closed loop from "I had an idea" to "the thing is on my desk doing the thing," with the 3D learning surface eating the "what is this and why" gap that simulators don't even try to fill.

## What Makes This Cool

**The 3D component library doubles as a teacher.** Every glTF model has clickable hotspots. Click the photoresistor: "This changes resistance with light. The Arduino reads that change and turns it into a number." Click pin GPIO 18 on the Pico: "Connect this to the OLED's data line." Nobody else ships that. PleaseDontCode (pleasedontcode.com — the closest competitor: in-browser tool that takes a natural-language prompt and produces an Arduino sketch + wiring schematic + browser flash for ~100 boards) shipped the AI-to-flash loop; Arduino's own AI assistant wrote the code; nobody combined those with a parts-buying workflow and an interactive component encyclopedia for a person who's never wired a breadboard.

The other piece nobody else has: **scope as a feature**. Competitors brag about supporting 100+ boards. Volteux supports 4 boards × 5 archetypes — and that constraint is the entire reason it can be reliable for a beginner. Within those 25 component SKUs and 5 well-understood patterns, the LLM almost can't go wrong; outside them, you fail loudly and honestly with an Honest Gap message rather than fudging.

## Constraints

- **Browsers:** Chromium only (ESP Web Tools and File System Access API both require it).
- **Cost:** Free to users. Hobby project. Only revenue is Adafruit affiliate kickback via AvantLink — and even that is out of scope for v1 distribution.
- **Team:** 2 people. Parallelizable via JSON contract.
- **Boards (4):** Uno, ESP32-WROOM, ESP32-C3, Pi Pico (Arduino-pico core).
- **Archetypes (5):** ultrasonic+servo, audio dashboard, OLED menu, DHT+Adafruit IO, photoresistor LED strip.
- **Components:** ~25 Adafruit SKUs, sized to archetypes.
- **Code:** Arduino C++ across all boards. One editor, one compiler, one evaluator.
- **No mains voltage, motor drivers >1A, LiPo charging, or anything safety-critical.**

## Premises (agreed)

1. **Wedge:** Volteux's defensible value is the idea-to-physical-hardware loop plus interactive 3D learning, not the AI code generation. PleaseDontCode beats us on board coverage; we beat them on "I have never touched a breadboard."
2. **Audience focus:** Beginner-first. Maker can use it but every hard call goes the beginner's way.
3. **Scope as feature:** 4 boards × 5 archetypes is the product, not a v1 limitation.
4. **3D view is the demo.** Designed, not deferred. The architecture for component metadata + click handlers exists from day one even if the visual fidelity ships rough.
5. ~~Distribution~~ — out of scope for v1. Build the product first; reach is a future problem.
6. **Pi Pico stays.** Design around the BOOTSEL UX with the File System Access API: after BOOTSEL puts the Pico in mass-storage mode, the page asks the user to pick the `RPI-RP2` drive via `showDirectoryPicker()`, then writes the UF2 silently. **One picker dialog per BOOTSEL cycle.** Re-flashing means BOOTSELing again, which re-mounts the volume and re-prompts. Don't try to cache permission across remounts — the volume identity changes. Honest framing: three physical steps (unplug, hold BOOTSEL, plug in) plus one folder-picker dialog per flash. Zero command-line, zero `cp` from a Finder window.

## Boards × Archetypes Matrix

Each archetype maps to exactly one board. This is classification, not free choice.

| Archetype ID | Board | Why this board |
|---|---|---|
| `uno-ultrasonic-servo` | Uno | Cheapest beginner board; 5V tolerant for HC-SR04 echo pin without level shifting. |
| `esp32-audio-dashboard` | ESP32-WROOM | Needs WiFi + enough flash for LittleFS-served HTML. |
| `pico-rotary-oled` | Pi Pico | Cheap, beginner-loved, plenty of GPIO for encoder + I2C. |
| `esp32c3-dht-aio` | ESP32-C3 | Cheap WiFi MCU; small footprint for compact builds. |

## Pipeline Architecture

```
User prompt
    ↓
[Intent classifier]  →  out-of-scope?  →  Honest Gap
    ↓
[LLM with structured output]  →  JSON document
    ↓
[Gate 1: Schema]   ────fail──→  retry once with error context  →  fail twice → Honest Gap
    ↓
[Gate 2: arduino-cli compile]   ────fail──→  retry once with stderr  →  fail twice → Honest Gap
    ↓
[Gate 3: Rules engine]   ────red──→  Honest Gap (+ amber/blue surfaced as warnings)
    ↓
[Gate 4: Cross-consistency]   ────fail──→  Honest Gap
    ↓
Validated JSON  →  shipped to UI
```

**Where arduino-cli runs:** A small always-on VPS, not serverless. Cloudflare Workers and Vercel functions can't host the AVR / ESP / Pico toolchains — no native execution, no persistent filesystem, no `gcc`. **v0 sizing**: ~4GB RAM, 30GB disk, 2 vCPU. The CX11 (2GB) won't fit even just `avr-gcc` plus `arduino-cli` cores once Servo, NeoPixel, etc. are pre-installed. Hetzner CX21 (~€6/mo, 4GB) or Fly.io shared-cpu-2x (4GB, ~$5/mo) is the floor. **v1.5 sizing**: when ESP and Pico toolchains land, peak compile RAM hits ~1.5-2GB and the disk image grows to 8-12GB; bump to ~8GB RAM / 50GB disk or split into per-board containers.

**Compile API contract.**
```
POST /api/compile
Headers: { Authorization: "Bearer <session-token>" }
Body:    { fqbn, sketch_main_ino, additional_files?, libraries: [allowlisted names] }
Response: { ok: bool, stderr: string, artifact_b64?: string, artifact_kind: "hex" | "uf2" | "bin" }
```
Auth/rate limit: a per-IP token bucket (10 compiles / 5 min) plus a Turnstile challenge before the demo URL is ever shared. v0 accepts the abuse risk because the URL is unlisted; revisit before any public link.

**Library allowlist.** A closed set of libraries pre-installed on the VPS at image-build time. Reject any sketch whose `libraries[]` references something outside the allowlist before invoking the compiler. v0 allowlist (one or two per archetype):
- archetype 1: `Servo`
- archetype 2 (v1.5): `WiFiManager`, `ESPAsyncWebServer`, `LittleFS_esp32`, `ArduinoJson`
- archetype 3 (v1.5): `Adafruit_SSD1306`, `Adafruit_GFX`, `Encoder`
- archetype 4 (v1.5): `WiFiManager`, `Adafruit_MQTT_Library`, `DHT_sensor_library`
- archetype 5 (v1.5): `Adafruit_NeoPixel`

**LLM call location.** Server-side, using the team's API key, for the v0 friend-demo. A friend should not have to bring their own key. The "client-side with user-provided key" option is a v1.5 self-serve mode for users who want to BYO key for cost reasons.

**Compile cache.** Cache key: `SHA256(sketch.main_ino + libraries.sort().join(','))`. Cache stores the compiled binary, the compile timestamp, and the originating archetype. TTL 30 days. On cache hit, the compile API returns the binary directly without invoking arduino-cli (~50ms vs ~3-8s). On VPS restart, only the canonical fixtures are precompiled; the rest is lazy. Cache hit rate target on a demo: ≥ 80% (kids retry near-identical prompts).

**Eval CI policy.** The full eval suite runs in CI on every PR that touches `pipeline/prompts/`, `rules/`, `meta/`, or `schemas/`. The PR fails if any of the 4-axis scores drop below baseline thresholds (see Success Criteria). Wall time impact: ~3-5 min added to those PRs. Cost impact: ~$5-15/week. Other PRs (UI work, frontend, fixtures) skip the eval suite.

**Latency UX.** End-to-end first-paint of all four views runs ~9-20s (intent classify → LLM generate → schema → compile → rules). The frontend must show a multi-step progress indicator ("Classifying your idea..." → "Writing your sketch..." → "Compiling for Arduino..." → "Checking the wiring..." → "Done") so the wait reads as considered, not broken.

## Information Architecture (post-generation main app view)

**Dominance story:** the 3D learning surface IS the wedge. Layout backs that up.

```
+------------------------------------------------------------------+
|  Volteux                          [prompt as chip]   [Uno • 92%] |  ← header
+------------------------------------------------------------------+
|                                          |                       |
|                                          |   2D BREADBOARD       |
|                                          |   (SVG, hole-addr)    |
|     3D INTERACTIVE COMPONENT VIEW        |                       |
|     (~55-60% width, hero panel,          +-----------------------+
|      click-to-label hotspots)            |   SKETCH EDITOR       |
|                                          |   (Monaco, read-only) |
|                                          |                       |
|                                          +-----------------------+
|                                          |   PARTS LIST          |
|                                          |   [Buy on Adafruit]   |
+------------------------------------------------------------------+
|                                                  [Flash to Uno]  |  ← primary CTA
+------------------------------------------------------------------+
```

- **Hero (60%):** 3D interactive component view, left half. This is the wedge. It loads first, paints first, gets the most pixels.
- **Right column (40%):** breadboard, sketch, parts list stacked. Each panel is supporting context, not equal-billed. Each can be expanded full-width on click but collapses back when 3D scene needs focus.
- **Header:** wordmark left, prompt chip middle (editable to retry), classification chip right.
- **Primary CTA:** "Flash to my Uno" anchored bottom right, only visible after generation succeeds. Disappears in Honest Gap and error states.
- **Empty state (before any prompt):** centered prompt input with 3-5 tappable example chips ("a robot that waves when something gets close", "a light that brightens when the room gets dark", "a temperature display that texts me when the room is too hot"). No empty 3D scene visible — the whole below-header area is the prompt-and-examples surface until the first generation.

**The hierarchy says:** "Volteux is THE 3D Arduino learning tool that also writes the code and tells you what to buy." Not: "Volteux is a generic Arduino dashboard with a 3D widget."

**v1.5 perf watchpoints:**
- 25 components in the 3D scene: cap polycount per component at 5000 tris, batch geometry, lazy-load via `<Suspense>`.
- **URL hash size for archetypes 2 and 4 will overflow.** Archetype 2 includes LittleFS HTML/CSS/JS for the dashboard; archetype 4 includes captive-portal HTML. base64+gzip pushes those to 8-15KB. Slack truncates URLs at ~2KB; iMessage at ~4KB; many mobile share sheets truncate aggressively. **The persistence story is broken on arrival for archetypes 2/4.** Resolution before v1.5: opt-in shareable-link service (server-side blob, short URL like `volteux.app/p/abc123`) for documents over 2KB. Auto-fallback for small projects continues using URL hash.

**Persistence.** v0: document persisted in the URL hash as base64(gzip(json)) **plus a server-side shareable-link service** (`POST /api/share` → returns `volteux.app/p/abc123` short URL backed by Cloudflare R2 or SQLite blob storage). The frontend falls back to the short-URL service automatically when the hash exceeds ~1800 bytes. Per-IP rate limit on `/share` (5/min) plus the same Turnstile + session token used by `/compile`. localStorage persists the latest project for browser-restart resilience. No server account in v0. v1.5: optional sign-in for a project library and named saves.

**Error boundary (generic).** Any failure in the LLM call, compile API, WebUSB flash, or external link surfaces through one shared UI component:

```
┌────────────────────────────────────────┐
│  Something didn't work.                │
│                                        │
│  [one-line description of what failed] │
│                                        │
│  [ Try again ]   [ Tell us what       │
│                    happened ]          │
└────────────────────────────────────────┘
```

- The "Try again" button retries the last action with the same parameters

- All errors are logged to a Sentry-style event sink (or just `console.error` in v0) with: action name, error class, stack trace.
- Case-specific copy and recovery flows are deferred to implementation discretion. The bar: a friend on demo day must always have a "what do I do now" answer beyond closing the tab.

## Definitions

**Honest Gap.** A user-facing message returned when the pipeline cannot reliably produce output. Triggered by exactly one of:
- (a) Intent classifier rejects the prompt (no archetype matches above confidence threshold), OR
- (b) Classified archetype but a required component/capability is out of scope (e.g., user wants HX711 + load cell), OR
- (c) Gate 1 (schema) fails twice (one auto-repair retry exhausted), OR
- (d) Gate 2 (compile) fails twice (one auto-repair retry exhausted), OR
- (e) Gate 3 (rules) returns red, OR
- (f) Gate 4 (cross-consistency) fails.

Format: "I can build the wiring and parts list for X, but I can't reliably write the Y portion. Want the partial version?" Never silent fudging, never made-up components.

**Type contract:** every surface that produces an Honest Gap (intent classifier, rules engine, cross-consistency gate) emits the same shape: `{ scope: 'full' | 'partial' | 'out-of-scope', missing_capabilities: string[], explanation: string }`. Each surface writes its own `explanation` copy. This is intentional v0 — copy consolidation deferred to v1.5 once we've seen which surfaces actually drift in tone. If drift gets bad, lift to a centralized `copy/honest-gap.ts` keyed file (i18n-shaped pattern).

**Cross-consistency gate.** Validates document referential integrity:
- Every `components[].id` is unique
- Every `connections[].from.component_id` and `connections[].to.component_id` exists in `components[]`
- Every `pin_label` in a connection exists in the source component's `pin_metadata`
- Every `breadboard_layout.components[].component_id` exists in `components[]`
- Every non-wire component (`type` in `sensor | actuator | display | passive | mcu`) has a corresponding `breadboard_layout.components[]` entry
- `board.fqbn` is one of the four allowed values
- Every Adafruit SKU in `components[]` resolves against the local SKU registry
- Every library in `sketch.libraries` is in the allowlist for this archetype

## Approaches Considered

### Approach A: One archetype, end to end first
Hard-code one archetype's JSON, build all four views and the cart and flash for it, *then* add the LLM. Fast to demo. Defers the natural-language magic. Doesn't fit a 2-person team (no parallelism).

### Approach B: Pipeline first, eval-driven, UI later
Build the LLM → JSON → schema → arduino-cli → rules pipeline as a CLI. Get compile-pass rate to 90% on the eval set. Then build views. Most rigorous; slowest to visual demo. Risks motivation. Doesn't fit a 2-person team for the same reason.

### Approach C: 3D component library first, AI generation later
Build the interactive 3D library first; AI follows. Builds the wedge first. Asset-heavy.

### Approach D (chosen): Contract-first parallel build
Day 1 the team locks the JSON schema together. UI track does Approach C (3D-first, components as the foundation). Pipeline track does Approach B (eval-driven). Both work in parallel against shared fixtures. Integration in week 4-6.

## Recommended Approach

**Approach D, with the following structure.**

### Day 1 (joint, ~half day)

Sit together. Write `schemas/document.schema.json`. Generate TypeScript types from it. Commit one canonical fixture: `fixtures/uno-ultrasonic-servo.json` — a hand-crafted document for archetype 1. Both sides develop against fixtures from this point.

**Note on board-as-component.** The fixture intentionally lists the Uno as both `board` (top-level metadata for the FQBN and SKU) and a `components[]` entry with `id: "u1"` (so the breadboard view and connections graph can reference it as a node). The pipeline produces this shape. The UI consumes this shape. Schema changes require both signatures and a `schemas/CHANGELOG.md` entry.

### Single source of truth: `components/registry.ts`

Static component metadata (name, type, `education_blurb`, `model_url`, `pin_metadata` including `anchor` and `pin_layout`) lives in **one TypeScript file**: `components/registry.ts`. Pipeline, UI, rules engine, and 3D library all import from it. **This is non-negotiable** — it's the DRY violation most likely to silently break the system in v1.5.

Consequence for the schema: the runtime JSON's `components[]` entries shrink. They carry only the *per-instance runtime data*:

```json
{
  "id": "s1",
  "sku": "3942",
  "quantity": 1
}
```

When the UI receives `{ id: "s1", sku: "3942", quantity: 1 }` it dereferences `registry["3942"]` to get name, type, education_blurb, pin_metadata, etc. The pipeline is also forbidden from inventing new SKUs — the cross-consistency gate verifies every emitted SKU exists in the registry.

Registry shape (pseudocode):

```ts
// components/registry.ts
export const COMPONENTS = {
  "50": {
    name: "Arduino Uno R3",
    type: "mcu",
    education_blurb: "The brain. It runs your sketch and reads/writes the pins.",
    model_url: "/models/uno-r3.glb",
    pin_metadata: [/* ... */],
    pin_layout: [/* ... */],
  },
  "3942": { name: "HC-SR04 ultrasonic sensor", /* ... */ },
  "169":  { name: "Micro servo SG90", /* ... */ },
  // ... 25 entries at v1.5
} as const;
```

Rules engine, eval cases, and the LLM prompt all reference SKUs by string. The registry is the source — anywhere else that names a component is consuming, never authoritative.

### Meta-Harness loop (v0)

```
            ┌─────────────────────────────────────┐
            │  Eval set (30-50 prompt → expected) │
            └────────────────┬────────────────────┘
                             │
                  ┌──────────▼──────────┐
                  │   Run pipeline      │
                  │  (schema, compile,  │
                  │   rules, classify)  │
                  └──────────┬──────────┘
                             │
                ┌────────────┴────────────┐
                ▼                          ▼
        ┌──────────────┐          ┌──────────────────┐
        │  Scores      │          │  Execution       │
        │  per case    │          │  traces (JSON):  │
        │  (4-axis)    │          │  gate outcomes,  │
        └──────┬───────┘          │  retries, errors │
               │                  └─────────┬────────┘
               │                            │
               └─────────┬──────────────────┘
                         │
                ┌────────▼──────────────┐
                │  Agentic Proposer     │
                │  (Claude Sonnet 4.6)  │
                │  reads:               │
                │  - prompt source      │
                │  - schema source      │
                │  - rule definitions   │
                │  - all scores         │
                │  - all traces         │
                │  - prior proposals    │
                │                       │
                │  outputs: candidate   │
                │  prompt/rule edits    │
                └──────────┬────────────┘
                           │
                ┌──────────▼─────────┐
                │  Human reviews     │
                │  proposal in PR    │
                │  (or auto-merge    │
                │  with rollback if  │
                │  scores improve)   │
                └──────────┬─────────┘
                           │
                ┌──────────▼─────────┐
                │  Re-run eval set,  │
                │  update baselines  │
                └────────────────────┘
```

The proposer's job is to improve the *harness* — the prompts, the schema, the rule definitions — not the LLM weights. It reads what the harness currently looks like, what it's currently scoring, and where it's currently failing, then proposes edits. The human (or an auto-merge guard with rollback) decides what lands.

**Implementation footprint:**
- `eval/cases/archetype-1/*.json` — eval cases with expected behaviors
- `eval/run.ts` — runs pipeline against cases, writes scores + traces to `eval/runs/<timestamp>/`
- `meta/proposer.ts` — agentic proposer; reads `eval/runs/`, `pipeline/prompts/`, `pipeline/schema/`, `rules/`; writes proposed-edits to `meta/proposals/<timestamp>/`
- `meta/review.ts` — applies proposal, re-runs eval, compares scores; auto-rollback if regression > X%

**Honest framing on the meta-harness from the paper:** the paper's evaluation is on text classification, math reasoning, and TerminalBench-2 — NOT hardware code generation. We're betting the architecture transfers. If the proposer's edits don't actually move the score on Volteux's eval set after 3-4 cycles, drop the proposer and keep the eval harness — it's still useful as a regression net.

## Visual Design — Unresolved Decisions

The visual identity of Volteux is deferred to a later design session. Three reference variants were generated and reviewed; none was adopted. They live at `~/.gstack/projects/taliamekh-Volteux/designs/main-app-view-20260425/` for context but are not the chosen direction.

The information architecture and interaction state spec above are locked. The decisions below need to be made by whoever designs the visual layer (later you, or a follow-up Claude session reading this doc).

### Brand & visual identity
- **Wordmark style:** is "Volteux" set in a distinctive serif (editorial), a geometric sans (Linear-style), or a custom mark? Affects every header, share-link footer, and printed export.
- **Accent color:** one accent only, not blue, not purple. Candidates from the variants: deep coral red, warm orange, electric green. **NOT** a gradient. Decide and add to `DESIGN.md` as a CSS variable.
- **Typography pairing:** display + body, two families max. Specifically NOT Inter/Roboto/Arial. Candidates: IBM Plex Serif + IBM Plex Sans, GT Sectra + Söhne, or Söhne alone. Pick before writing any CSS.
- **Surface treatment:** flat, hairline borders, or subtle elevation? The interaction state spec assumes hairline borders + small drop shadows on hover. Confirm.
- **3D scene aesthetic:** "real product photography" (variant C-style) or "isometric line drawing" (variant B-style) or "stylized PBR" (variant A-style)? This is the wedge — high-stakes call. Real product photography is the most aspirational; stylized PBR is the easiest to render at 60fps with React-Three-Fiber.

### Layout specifics
- **Header height:** a thin 48px chrome strip, or a more substantial 80-96px brand bar?
- **Right-column panel split:** the breadboard, sketch, and parts list stack vertically — what are the height ratios? The spec implies breadboard ≥ sketch ≥ parts list but doesn't specify exact splits.
- **Resizable panels?** Should the user be able to drag the divider between the 3D hero and the right column? Useful for power users; complexity for beginners. Default off, ship a v1.5 setting?
- **3D camera controls:** orbit (drag to rotate), pan, zoom — which are exposed? Recommendation: orbit + zoom only, no pan, with a "reset view" button visible in a corner.
- **Empty-state preview:** when no project exists, does the page show ONLY the prompt input + examples, or does it also show a static teaser 3D scene of "what you could build"? A teaser sells the wedge faster but costs render budget on first paint.

### Copy & tone
- **Wordmark and tagline:** "Volteux" alone, or with a tagline? Candidate: "Build the thing." Or none. Subtagline is allowed in the share preview only.
- **Example prompts on empty state:** spec says 3-5 chips. Author them — they need to be specific, not generic. Candidates: "a robot that waves when something gets close", "a light that brightens when the room gets dark", "a temperature display that texts me when the room gets too hot". Pick 3-5 in your voice.
- **Honest Gap voice:** the schema lets each surface (intent classifier, rules engine, cross-consistency) write its own copy, but the audience is a 14-year-old. Define the tone (warm, direct, never condescending, never apologetic) so all three surfaces sound like the same product.
- **Step labels in the multi-step progress:** "Classifying your idea..." is a placeholder. Pick the actual copy. Same for "Writing your sketch...", "Compiling for Arduino...", "Checking the wiring..." Each label is the user's only signal that the system is doing real work.
- **Flash-success copy:** "Your Uno is running your code." is a placeholder. Make it land for a 14-year-old who just got their first hardware to do a thing. This is the magic moment.

### Iconography & decoration
- **Icons in the parts list:** photographic thumbnails of each component (Adafruit photos), simplified vector icons, or no icons? Recommendation: small photographic thumbnails — uses the Adafruit photos you've already approved for textures, ties parts list to 3D scene visually.
- **No icons in colored circles.** If you draft a layout that has them, that's the AI-slop tell.
- **Status indicators in the multi-step progress:** what does the "active" state look like vs "complete" vs "queued"? Specifically NOT a generic spinner everywhere. Each step earns its own visual.

### 3D component scene specifics
- **Lighting:** soft studio (one key + ambient) or harsher single-source for drama? Affects how the components read on click.
- **Component rotation when clicked:** does the camera ease toward the clicked component, or stay put? Recommendation: subtle ease-in (300ms) to gently focus attention; full camera lock-in is too aggressive.
- **Pin label callout style:** floating tooltip, side panel, or inline label over the component? Each implies a different interaction model. The state spec assumes floating tooltip; confirm or override.
- **What happens when a beginner doesn't know to click?** First-visit hint state: pulse on the first component, "click to learn more" copy that fades after 5s. Confirm the interaction or pick something stronger (full overlay tutorial on first visit).

### Accessibility (must not be deferred)
- **Keyboard navigation through 3D scene:** Tab cycles through components in a defined order; Enter selects; Escape clears. The 3D view is focus-trapped or focus-leak-able? Decide and document.
- **Screen reader for 3D component selection:** when a user selects a component via keyboard, the education_blurb must be announced. Implement via ARIA live region.
- **Color contrast:** the accent color you choose must hit WCAG AA contrast against the background for all text and icons that USE it. Verify before locking in.
- **Touch targets:** desktop-only spec, but tablets in landscape will hit this. 44px min for any tap target. The component-click hotspots in 3D need to be at least 44px in screen space at the default camera distance.
